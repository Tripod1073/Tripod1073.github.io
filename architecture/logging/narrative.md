# Centralized Logging Architecture Narrative

## Purpose

This document provides a detailed description of the centralized logging architecture used to collect, protect, and retain security-relevant logs across AWS accounts.

The architecture ensures that logs required for security monitoring, forensic analysis, and compliance reporting are delivered to a centralized security account and stored in immutable, encrypted storage.

This document expands on the high-level overview and describes the operational and security characteristics of the logging system.

---

# Architecture goals

The logging architecture was designed to meet the following goals:

• Centralize security logs from multiple AWS accounts  
• Protect logs from modification or deletion  
• Ensure logs are encrypted at rest and in transit  
• Support monitoring and alerting for logging failures  
• Maintain log retention consistent with regulatory requirements  
• Enable independent security review and forensic investigation  

These goals align with common control requirements found in:

• NIST SP 800-53 audit and accountability controls  
• FedRAMP logging and monitoring requirements  
• CMMC audit logging practices  
• ISO 27001 monitoring and event logging controls  

---

# Architecture components

The centralized logging architecture consists of several primary components.

## Log-producing services

Multiple AWS services generate logs that must be retained for security and operational purposes.

Examples include:

• CloudTrail management and data events  
• VPC Flow Logs  
• AWS service logs (WAF, ELB, API Gateway, etc.)  
• Application-level logs generated by deployed services  

Each service produces logs within its originating account.

---

## Log delivery mechanisms

Logs are transmitted from producing accounts to the centralized logging environment through controlled delivery mechanisms.

Common mechanisms include:

• AWS CloudTrail organization trails  
• CloudWatch Logs subscription filters  
• Kinesis Data Firehose delivery streams  
• Direct S3 log delivery integrations  

These delivery paths ensure that logs are transferred securely and consistently to the centralized logging account.

---

## Central security logging account

All security-relevant logs are aggregated in a dedicated security logging account.

Separating log storage into an independent account reduces the risk that an attacker who compromises a workload account could modify or delete historical logs.

The security account is responsible for:

• long-term log storage  
• log protection controls  
• encryption key management  
• monitoring and alerting  

Access to this account is restricted to authorized security personnel and automated logging services.

---

## Immutable log storage

Logs are stored in Amazon S3 buckets configured with Object Lock in compliance mode.

Object Lock prevents modification or deletion of stored objects during the configured retention period.

Key properties include:

• write-once-read-many behavior  
• retention enforcement at the storage layer  
• protection against accidental deletion  
• protection against malicious tampering  

Object Lock ensures that once logs are delivered, they cannot be altered.

---

## Encryption and key management

All logs stored in S3 are encrypted using AWS Key Management Service (KMS).

Encryption ensures that log data is protected against unauthorized access while stored at rest.

Key management responsibilities include:

• defining key policies restricting usage to logging services  
• controlling administrative access to encryption keys  
• enabling key rotation according to organizational policy  

KMS encryption integrates directly with S3 to provide automatic encryption for newly stored objects.

---

## Application log handling

Application logs are handled differently from security telemetry.

While security logs are centralized in the security logging account, **application logs remain within their originating workload accounts**.

This design decision ensures that sensitive application data is not unnecessarily exposed outside of the workload boundary.

Typical application log content may include:

• request metadata  
• debugging information  
• application errors  
• operational telemetry  

In some environments these logs may contain:

• customer identifiers  
• transaction data  
• internal system state  

Centralizing this data across accounts could create unnecessary access exposure.

---

## Workload account log storage

Each workload account maintains its own application log storage bucket.

These buckets are configured to enforce protections similar to the centralized security log store.

Typical controls include:

• S3 Object Lock for immutability  
• server-side encryption using KMS  
• restricted administrative access  
• defined retention policies  

This approach preserves the integrity of application logs while maintaining appropriate data isolation.

---

## Separation of telemetry types

The architecture separates logging into two categories.

| Log Type | Storage Location | Purpose |
|----------|------------------|---------|
| Security telemetry | Central security logging account | Detection, monitoring, audit |
| Application logs | Workload accounts | Debugging, operations, application analysis |

Security-relevant telemetry such as CloudTrail and infrastructure logs are centralized to support monitoring and incident response.

Application-generated logs remain local to their workload environments to reduce exposure of potentially sensitive data.

---

## Investigative access to application logs

When necessary, security investigators may require access to application logs during incident response.

Access to these logs is not centralized automatically. Instead, investigators obtain temporary access through established access control procedures within the workload account.

This approach preserves data isolation while still enabling security investigations when required.

---

# Log protection strategy

The architecture includes several safeguards designed to protect the integrity and availability of logging data.

## Cross-account isolation

Logs are stored in a dedicated security account rather than the originating workload accounts.

This prevents attackers who compromise a workload environment from directly manipulating stored logs.

---

## Immutable storage enforcement

Object Lock ensures that logs cannot be deleted or overwritten during the defined retention window.

This mechanism protects against:

• insider threats  
• compromised administrative credentials  
• automated deletion attempts  

---

## Restricted administrative access

Administrative access to logging infrastructure is tightly controlled.

Typical restrictions include:

• limited administrative roles  
• strong authentication requirements  
• separation of operational and security roles  

These controls reduce the likelihood of unauthorized changes to the logging system.

---

# Monitoring and failure detection

The logging architecture includes mechanisms to detect failures in log generation or delivery.

Monitoring systems observe:

• CloudTrail delivery status  
• Firehose delivery failures  
• CloudWatch log ingestion errors  
• S3 bucket access anomalies  

Alerts are generated when failures are detected so that security teams can respond quickly.

---

# Operational considerations

Operating a centralized logging system requires ongoing monitoring and periodic review.

Operational tasks may include:

• verifying log delivery health  
• reviewing retention policies  
• rotating encryption keys  
• validating cross-account trust relationships  

Regular operational reviews help ensure the logging architecture continues to function as designed.

# Summary

The centralized logging architecture provides a secure and reliable mechanism for collecting and protecting security logs across AWS accounts.

By combining cross-account log aggregation, immutable storage, encryption, and monitoring, the architecture ensures that logging data remains trustworthy and available for both operational monitoring and compliance verification.
